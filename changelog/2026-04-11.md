## Новое

- **Composition Picker на шаге расчёта** — если на один КП есть несколько КП поставщиков с пересекающимися позициями, продажи теперь могут выбрать по каждой позиции, от какого поставщика брать цену. Расчёт использует выбранное сочетание цен. Picker появляется только когда реально есть выбор (два и более поставщиков на одну позицию); для обычных одно-поставщиковых КП ничего не меняется
- **Bypass при создании КП поставщика** — если создаёшь второй КП от того же поставщика на том же запросе, поля `pickup_country` / `pickup_city` автоматически наследуются от первого КП (не нужно вводить заново). Для нового поставщика остаётся прежнее поведение — pickup_country подтягивается из карточки поставщика
- **Верификация КП поставщика** — procurement может явно пометить КП как верифицированный (готовый к композиции). Прямые правки верифицированного КП теперь требуют согласования head_of_procurement через новое модальное окно с diff-таблицей и обоснованием

## Технические детали

### Database

- **Миграция 263** — новая таблица `kvota.invoice_item_prices` (junction между `invoices` и `quote_items`) с колонками: per-item price fields, versioning (`version`, `frozen_at`, `frozen_by`), organization_id для RLS
- **Миграция 264** — `quote_items.composition_selected_invoice_id` (указатель на активную позицию в composition) + `invoices.verified_at` / `verified_by` (верификация)
- **Миграция 265** — идемпотентный backfill: iip rows для всех существующих (quote_item, invoice) пар, composition pointer = invoice_id, invoices.verified_at для `status='completed'`
- **RLS** на `invoice_item_prices` — org + role pattern (10 SELECT ролей, 4 WRITE, 2 DELETE). Исправление после pre-check обнаружения что `kvota.invoices` сам без RLS — оригинальный "mirror invoices" подход не работал
- Верифицированный bit-identity алгебраически доказан на всех 124 backfilled items: `COALESCE(qi.purchase_price_original, qi.base_price_vat)` = `COALESCE(iip.purchase_price_original, qi.base_price_vat)` → calculation engine видит те же числа

### Python backend

- **Новый `services/composition_service.py`** — pure сервис из 5 функций: `get_composed_items` (адаптер для calc pipeline), `get_composition_view` (для API), `apply_composition`, `validate_composition`, `freeze_composition`
- **Hook в main.py** — 3 call-сайта `quote_items` reads перед `build_calculation_inputs()` (lines ~13314, 14003, 14689) заменены на `composition_service.get_composed_items(quote_id, supabase)`. `calculation_engine.py` / `calculation_models.py` / `calculation_mapper.py` не тронуты (проверено git diff'ом)
- **Новый `api/composition.py`** — 6 эндпоинтов:
  - `GET  /api/quotes/{id}/composition`
  - `POST /api/quotes/{id}/composition`
  - `POST /api/invoices/{id}/verify`
  - `POST /api/invoices/{id}/edit-request`
  - `POST /api/invoices/{id}/edit-approval/{approval_id}/approve`
  - `POST /api/invoices/{id}/edit-approval/{approval_id}/reject`
- Invoice-edit approval flow переиспользует таблицу `kvota.approvals` с `approval_type='invoice_edit'` и diff в `modifications` JSONB. Не использует `approval_service.request_approval()` — та функция жёстко привязана к quote→top_manager workflow

### Next.js frontend

- **`useCompositionPicker`** — новый client component в `features/quotes/ui/calculation-step/composition-picker.tsx`, отрендерился в `CalculationStep` между формой и результатами
- **Bypass logic в `createInvoice`** — sibling check, `bypass_reason` в return type, inherited pickup fields
- **iip row insertion в `assignItemsToInvoice`** — 2-query fetch (quote_items + quotes), upsert с ignoreDuplicates
- **`EditVerifiedRequestModal`** — standalone модальное окно, готовое к wiring в invoice-edit экран. Shadcn Dialog + Textarea, diff-таблица, валидация reason (min 10 chars)
- **`canEditComposition` role helper** в `shared/lib/roles.ts` — зеркалит backend'ный `COMPOSITION_WRITE_ROLES`

## Миграция

Миграции 263/264/265 — чисто additive, ничего не ломают. Все pre-Phase-5b КП продолжают рассчитываться идентично (bit-identity proven on prod data).

Для workflow остался прежний edit path для не-верифицированных КП поставщика — только верифицированные требуют согласования.
