## Новое

### Procurement Phase 3

- **Страна отгрузки на КП поставщика** — в форме создания КП поставщику появился выбор страны отгрузки (двуязычный список RU+EN, 264 страны). Страна подхватывается автоматически из карточки поставщика и сохраняется как ISO-код. Нужно для автоматического расчёта НДС в следующем релизе
- **Полный список Инкотермс 2020** — везде, где указывались условия поставки, теперь доступны все 11 терминов: EXW, FCA, CPT, CIP, DAP, **DPU**, DDP, FAS, FOB, CFR, CIF (раньше было только 5). Добавлено в создание КП клиенту, создание КП поставщику и форму расчёта
- **Мин. заказ (MOQ) на позициях КП поставщика** — в таблице поставщика появилась колонка «Мин. заказ» между «Кол» и «Цена». Когда количество меньше минимального заказа поставщика, позиция подсвечивается жёлтым с подсказкой, а в шапке показывается счётчик нарушений. Ограничение мягкое — расчёт не блокируется
- **5 новых валют** — к USD/EUR/RUB/CNY/TRY добавились AED (дирхам ОАЭ), KZT (тенге), JPY (йена), GBP (фунт) и CHF (швейцарский франк). Все поддерживаются ЦБ РФ, работают во всех формах расчёта, логистики, КП поставщика и платежей
- **Поиск городов через HERE работает стабильнее** — локальный кэш результатов на 256 записей снижает нагрузку на HERE API, поддержка всех 250 стран ISO вместо 28 жёстко зашитых (пропадала, например, Бразилия и Египет)

### Мульти-поставщик в КП (Phase 5b)

- **Composition Picker на шаге расчёта** — если на один КП есть несколько КП поставщиков с пересекающимися позициями, продажи теперь могут выбрать по каждой позиции, от какого поставщика брать цену. Расчёт использует выбранное сочетание цен. Picker появляется только когда реально есть выбор (два и более поставщиков на одну позицию); для обычных одно-поставщиковых КП ничего не меняется
- **Bypass при создании КП поставщика** — если создаёшь второй КП от того же поставщика на том же запросе, поля `pickup_country` / `pickup_city` автоматически наследуются от первого КП (не нужно вводить заново). Для нового поставщика остаётся прежнее поведение — pickup_country подтягивается из карточки поставщика
- **Верификация КП поставщика** — procurement может явно пометить КП как верифицированный (готовый к композиции). Прямые правки верифицированного КП теперь требуют согласования head_of_procurement через новое модальное окно с diff-таблицей и обоснованием

### Phase 5b — Quote Composition Engine

#### Database

- **Миграция 263** — новая таблица `kvota.invoice_item_prices` (junction между `invoices` и `quote_items`) с колонками: per-item price fields, versioning (`version`, `frozen_at`, `frozen_by`), organization_id для RLS
- **Миграция 264** — `quote_items.composition_selected_invoice_id` (указатель на активную позицию в composition) + `invoices.verified_at` / `verified_by` (верификация)
- **Миграция 265** — идемпотентный backfill: iip rows для всех существующих (quote_item, invoice) пар, composition pointer = invoice_id, invoices.verified_at для `status='completed'`
- **RLS** на `invoice_item_prices` — org + role pattern (10 SELECT ролей, 4 WRITE, 2 DELETE). Исправление после pre-check обнаружения что `kvota.invoices` сам без RLS — оригинальный "mirror invoices" подход не работал
- Бит-identity алгебраически доказана на всех 124 backfilled items: `COALESCE(qi.purchase_price_original, qi.base_price_vat)` = `COALESCE(iip.purchase_price_original, qi.base_price_vat)` → calculation engine видит те же числа

#### Python backend

- **Новый `services/composition_service.py`** — pure сервис из 5 функций: `get_composed_items` (адаптер для calc pipeline), `get_composition_view`, `apply_composition`, `validate_composition`, `freeze_composition`
- **Hook в main.py** — 3 call-сайта `quote_items` reads перед `build_calculation_inputs()` (lines ~13314, 14003, 14689) заменены на `composition_service.get_composed_items(quote_id, supabase)`. `calculation_engine.py` / `calculation_models.py` / `calculation_mapper.py` не тронуты (проверено git diff'ом + AST импорт-чеком)
- **Новый `api/composition.py`** — 6 эндпоинтов:
  - `GET  /api/quotes/{id}/composition`
  - `POST /api/quotes/{id}/composition`
  - `POST /api/invoices/{id}/verify`
  - `POST /api/invoices/{id}/edit-request`
  - `POST /api/invoices/{id}/edit-approval/{approval_id}/approve`
  - `POST /api/invoices/{id}/edit-approval/{approval_id}/reject`
- Invoice-edit approval flow переиспользует таблицу `kvota.approvals` с `approval_type='invoice_edit'` и diff в `modifications` JSONB. Не использует `approval_service.request_approval()` — та функция жёстко привязана к quote→top_manager workflow

#### Next.js frontend

- **`CompositionPicker`** — новый client component в `features/quotes/ui/calculation-step/composition-picker.tsx`, отрендерился в `CalculationStep` между формой и результатами. Auto-hides когда нет multi-supplier выбора
- **Bypass logic в `createInvoice`** — sibling check, `bypass_reason` в return type, inherited pickup fields от первого КП от того же поставщика. Объединено с Phase 3 `pickup_country_override` / `pickup_country_code` / `supplier_incoterms` — явный user pick из модалки всегда побеждает
- **iip row insertion в `assignItemsToInvoice`** — 2-query fetch (quote_items + quotes), upsert с ignoreDuplicates. Без этого picker не имел бы данных для отрисовки
- **`EditVerifiedRequestModal`** — standalone модальное окно, готовое к wiring в invoice-edit экран. Shadcn Dialog + Textarea, diff-таблица, валидация reason (min 10 chars)
- **`canEditComposition` role helper** в `shared/lib/roles.ts` — зеркалит backend'ный `COMPOSITION_WRITE_ROLES`
- **Composition view types** (`CompositionView`, `CompositionItem`, `CompositionAlternative`) в `entities/quote/types.ts`

## Миграция

Миграции 263-268 — чисто additive, ничего не ломают. Все pre-Phase-5b КП продолжают рассчитываться идентично (bit-identity доказана на прод-данных).

Для workflow остался прежний edit path для не-верифицированных КП поставщика — только верифицированные требуют согласования через head_of_procurement.
