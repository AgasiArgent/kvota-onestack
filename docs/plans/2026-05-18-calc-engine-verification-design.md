# Стратегия верификации Calculation Engine

**Дата:** 2026-05-18
**Статус:** дизайн утверждён — ожидает ревью спеки → реализация по `/lean-tdd`
**Связано:** Testing 2 (строки №36, №42, №48, №51–53)

## 1. Цель

Доказать, что Calculation Engine OneStack воспроизводит «старую форму расчёта»
(Excel) с отклонением **≤ 0.01%**, и доказать это **наглядно для нон-кодинг
пользователей** — не только тестами в CI.

При любом расхождении верификация обязана **однозначно локализовать** его на
одной из трёх поверхностей:

1. **INPUT-маппинг** — данные приложения → входы движка (`build_calculation_inputs()`).
2. **CALC ENGINE** — `calculation_engine.py` (доверенный, исторически надёжный).
3. **OUTPUT-маппинг** — результаты движка → артефакт сравнения.

## 2. Жёсткое ограничение

Движок **заморожен**. `calculation_engine.py`, `calculation_models.py`,
`calculation_mapper.py` **не изменять** (правило CLAUDE.md). Это верификация, не
модификация движка. Дефект, найденный внутри движка, **репортится**, а не
правится в рамках этой работы.

## 3. Контекст — текущее состояние

**Три поверхности — текущее покрытие:**

- `test_excel_comparison.py` (корень репо) — прототип: standalone-скрипт, НЕ в
  pytest/CI; использует СОБСТВЕННЫЙ маппинг входов (не продакшн
  `build_calculation_inputs`); хардкод на один эталонный файл, которого больше
  нет на диске; `range(16,21)` — максимум 5 позиций. Покрывает только движок, и
  то через непродакшн-путь.
- `services/export_validation_service.py` — comparison-экспорт: листы
  `API_Inputs` / `расчет` (переподвязанный) / `API_Results` / `Comparison`,
  diff% и подсветка 0.01%. Качественный, НО: вызывается только из
  `legacy-fasthtml/quote_detail_and_workflow.py` (мёртвый рантайм после
  Phase 6C); переподвязывает `расчет` под наши входы — кормит одинаковые входы
  обеим сторонам сравнения и **слеп к ошибкам INPUT-маппинга**.
- Продакшн-путь входов: `composition_service.get_composed_items()` (БД→items) →
  `build_calculation_inputs(items, variables)` (`services/calculation_helpers.py`)
  → `calculate_multiproduct_quote()`. Golden-master регресс-теста против
  Excel-эталона нет.

**Сигнал от тестера (Testing 2):** на расчётных строках №36/№42/№48 — «Сумма КП
не тянется», «Не тянет Цену/Сумму в валюте КП», «Данных нет» (логистика/сроки).
Подсказка во всех 3 эталонных строках: «ПРОВЕРИТЬ КОЭФФИЦИЕНТ НАЦЕНКИ ПРИ
ОТСРОЧКЕ ПЛАТЕЖА — чтобы совпадал с excel». Подтверждает: ошибки — на проводке
входов/выходов, не в движке.

## 5. Допуск

Для каждого сверяемого значения:

```
|engine − эталон| / max(|эталон|, ε) ≤ 1e-4
```

При |эталон| ниже порога ε (около нуля) — абсолютный критерий
`|engine − эталон| ≤ ε_abs`. Предложение: ε = 1e-6, ε_abs = 0.01 в валюте КП;
финально зафиксировать в конфиге теста.

Каждое денежное значение помечено валютой; тест ассертит и валюту, и величину.

**Правило сегментов (логистика и таможня).** В Kvotaflow логистика может быть
разбита на произвольное число сегментов; в эталонном Excel — всегда 3 плеча
(T/U/V). Поэтому посегментная/попозиционная сверка логистики и таможни
некорректна. Тест сверяет логистику и таможню **только на уровне итога** (сумма
по сегментам и позициям) с допуском ≤0.01%; посегментные/попозиционные значения
выводятся информационно, pass/fail по ним не выставляется.

## 6. Трек A — автоматический регресс (гейт CI)

Гибрид из двух механизмов.

### A1 — Автономный golden-master (ядро)

Поток на каждый эталон:

```
.xlsm → извлечение входов → items[] + variables{}    (продакшн-форма)
      → build_calculation_inputs(items, variables)    [ПРОДАКШН, calculation_helpers.py]
      → calculate_multiproduct_quote(calc_inputs)      [ПРОДАКШН, движок]
      → сверка каждого выхода с кэш-значениями .xlsm   (допуск ≤0.01%)
```

- **Не трогает БД** — быстрый, детерминированный, годится как блокирующий гейт CI.
- **Шим «.xlsm → items/variables»** воспроизводит форму, которую отдаёт
  `composition_service.get_composed_items()`, чтобы `build_calculation_inputs`
  получал продакшн-форму. Шим тупой (ячейка→ключ), без бизнес-логики, ревьюится
  один раз, общий для всех файлов.
- **Пиннинг «живых» входов.** `build_calculation_inputs` конвертирует валюты
  через `services.currency_service.convert_amount`. Тест подменяет
  `convert_amount` стабом, возвращающим курсы из самого .xlsm (колонка Q — курс
  на позицию). Аналогично — резолв пошлин пиннится на значения из .xlsm. Так
  расхождение относится к маппингу/движку, а не к дрейфу справочных данных.
- **Контрольные точки** (ассерты не только на финале — финал может скрыть две
  компенсирующие ошибки):
  1. Закупка: N16 (цена без НДС), P16 (после скидки), R16 (цена/ед в валюте КП), S16 (сумма позиции).
  2. Логистика: **только итог** (Σ V16 / V13) — посегментные T16/U16/V16
     ожидаемо расходятся (правило сегментов, §5), сверяются информационно.
  3. Таможня: **только итог** (Σ Y16/Z16) — посегментные значения информационно.
  4. COGS: AA16 (на ед.), AB16 (на позицию).
  5. **Финблок (прицел тестера):** AD16/AE16, прибыль AF16, ЛПР AG16, forex AH16,
     фин-агент AI16, financing_cost BA16/BB16 — наценка при отсрочке платежа.
  6. Цена продажи: AJ16, AK16, AL16 (с НДС).
  7. НДС: AN16, AO16, AP16.
  8. Транзит: AQ16 (для АМТЕЛ).
  9. Тоталы квоты: S13, AB13, AF13, AK13, AL13 и финансовые BH/BJ/BL.
- **Корпус:** все 4 эталона; АМТЕЛ — все 103 строки из .xlsm.
- **Вердикт:** тест падает, если хоть одна контрольная точка вне допуска;
  в отчёте — позиция, ячейка, эталон, движок, отклонение.

### A2 — Снапшот-тест composition_service

- Снимок `get_composed_items()` для `Q-TEST-P5B-01` (готовая regression-фикстура
  composition_service, уже в БД) и Q-202605-0011 → закоммиченная фикстура →
  ассерт стабильности структуры/значений.
- Ловит регресс БД→items, который A1 не видит (A1 использует свой шим вместо
  `composition_service`).
- Полную ≤0.01%-сверку «БД→движок→эталон» включаем позже — когда тестовые квоты
  доукомплектуют недостающими входами (логистика/сроки; см. №48). До тех пор
  A2 — структурный снапшот.

### Размещение

- Тесты в `tests/` (подхватываются `pytest.ini`, `testpaths = tests`).
- Прогон в CI (`.github/workflows/ci.yml`) — блокирующий гейт.
- Прототип `test_excel_comparison.py` (корень репо) удаляется — его роль
  полностью поглощает A1.

## 7. Golden-фикстуры

- Формат — **JSON**. Скрипт `scripts/refresh_golden.py` извлекает из каждого
  .xlsm входы и эталонные значения ячеек → `tests/golden/<name>.json`.
- Структура записи: `{ source_xlsm, quote_idn, quote_currency, fx_rates{},
  inputs:{ variables{}, products[] }, expected:{ products[]:{cell:{value,currency}},
  totals:{} } }`.
- Тест A1 читает только JSON — быстро, без openpyxl на каждом прогоне.
- .xlsm в `tests/golden/sources/` — аудируемый источник; `refresh_golden.py`
  запускается вручную при изменении корпуса; PR-диф JSON показывает изменение
  эталонных чисел.

## 8. Трек B — офлайн comparison Excel (доказательство для нон-кодеров)

- Скрипт `scripts/generate_calc_comparison.py` на каждую эталонную квоту строит .xlsx:
  - **Вердикт-баннер сверху:** «ВСЁ СХОДИТСЯ — макс. отклонение 0.003%» или
    «РАСХОЖДЕНИЕ — N ячеек вне допуска».
  - **Лист «Сравнение»:** строка = `позиция | ячейка | поле | engine |
    эталон .xlsm | абс.откл | %откл | PASS/FAIL`, цветовая подсветка FAIL.
  - **Лист «Входные данные»:** кол-во / цена / валюта / группировка по позициям —
    видимый слой входов для изоляции поверхности.
- Эталонная колонка = **собственные кэш-значения .xlsm** (независимые,
  статичные), НЕ `расчет`, переподвязанный под наши входы.
- Основа — `services/export_validation_service.py`: добавляем новый режим
  «эталон из .xlsm» рядом с существующим «переподвязка расчёта»; существующий
  режим не ломаем.
- Без живого эндпоинта в Next.js/FastAPI — отложен (см. §10).

## 9. Свойство-локализатор

- A1 PASS ⇒ INPUT-маппинг (`build_calculation_inputs`) и движок корректны ⇒ баги
  вида №42 («сумма не тянется») лежат НИЖЕ движка — в проводке/отображении.
- A1 FAIL ⇒ упавшая контрольная точка + поверхность указывают, где искать.
- Лист «Входные данные» Трека B делает входы видимыми ⇒ расхождение однозначно
  относится к input / engine / output.

## 13. Детали реализации — решаются в `/lean-tdd` (TDD)

Не «дыры» дизайна, а уточнения, корректнее закрываемые тестами:

- Точная карта колонок .xlsm на позицию/входы. `test_excel_comparison.py` даёт
  стартовую карту (входы D/E/G/J/K/L/M/O/Q/W/X/AC, выходы N..BD); для
  транзитного АМТЕЛ цена в колонке I, не J/K — карта может зависеть от типа
  сделки.
- Точная форма `items`/`variables`, которую ждёт `build_calculation_inputs` —
  свериться с реальным выводом `composition_service.get_composed_items()`.
- Тип возврата `calculate_multiproduct_quote` (список результатов vs
  `QuoteCalculationResult`) — свериться с `calculation_engine.py`.
- Механизм пиннинга `convert_amount` (monkeypatch фикстурой pytest).
- В файле НДС22-18 колонка базовой цены прочиталась пустой при первом скане —
  уточнить реальную колонку при написании extractor.

## 14. Известные ограничения харнесса (для Фазы 2)

Выявлены ревью; не блокируют выпуск, учесть при доработке:

- **rubli — финансовый блок ниже S16 почти не ассертится.** Из-за правила
  сегментов попозиционные `AA16..BA16` и тоталы `Y13/AB13/AF13/AK13/AL13` для
  rubli — `EXPECTED_SEGMENT_DIFFS`/`KNOWN_DIVERGENCES` без вердикта. Новый баг
  именно на весовой ветке rubli может проскользнуть. Риск ограничен
  (idemitsu/forma/amtel хард-ассертят тот же код движка). Фаза 2: ассертить
  попозиционные ячейки rubli против замороженного снапшота вывода движка (не
  эталона) — регресс-защита без нарушения правила сегментов.
- **`_to_pct` (`golden_support.py`) — эвристика по модулю.** `|f|≤1` умножается
  на 100, `|f|>1` пропускается; наценка/пошлина ≥100% будет прочитана неверно.
  Текущий корпус безопасен (наценки 0.14–0.21, пошлины 0–0.15). Фаза 2: привязать
  масштаб к семантическому признаку (кол. AC всегда доля), а не к величине.
