## Problem

Today OneStack assumes each `quote_items` row has exactly one supplier offer: `quote_items.invoice_id` is a 1:1 pointer to a `kvota.invoices` row (a "КП поставщика"). Reality is more complex:

- Procurement often gets competing offers from multiple suppliers for the same items.
- Sales wants to compose the final customer quote by picking the best supplier *per item*, not per invoice.
- Today they either manually consolidate offers into one invoice (loses supplier history) or create parallel quotes (loses single-source-of-truth).

## Goal

Let procurement collect multiple `invoices` (supplier KPs) against the same quote, with each invoice holding its own prices for the items it covers. Let sales pick, per item, which invoice's price to use. Feed the chosen combination into the existing calculation engine without touching the engine itself.

## Non-goals (Phase 5b scope boundary)

- **Automatic supplier scoring/selection** — sales picks manually.
- **Multi-currency conversion inside the picker** — display each offer in its native currency; conversion happens downstream at calculation.
- **Bulk "select all from supplier X"** — per-item picking only. Can add later.
- **Changes to the actual KP-send mechanism (email, PDF generation)** — out of scope. Only the *freeze* hook (Decision #4) is in scope.
- **Multi-tenant support for composition** — reuse existing quote-level access control; no new tenant concept.

## Locked design decisions (do not re-negotiate)

1. **Migration path.** `quote_items.invoice_id` stays as a legacy pointer. The new `invoice_item_prices` junction supplements it — does not replace it. Backfill populates the junction from existing `(quote_item, invoice_id)` pairs.

2. **Composition UI placement.** The picker lives inside the calculation step (`frontend/src/features/quotes/ui/calculation-step/calculation-step.tsx`), rendered as a new card *between* `CalculationForm` and `CalculationResults`, *above* the Calculate button (which lives in `calculation-action-bar.tsx:90-102`).

3. **Logistics vs customs granularity.** Logistics data stays per-invoice (each supplier has their own shipping). Customs data stays per-item (HS codes live on `quote_items.customs_code`, unchanged).

4. **KP versions frozen on send.** When a KP is sent, its `invoice_item_prices` rows get `frozen_at` stamped. Frozen rows are immutable; a subsequent edit creates a new version (`version` int bumps). The *send mechanism* itself (email, PDF) stays as-is — only the freeze hook is in scope for Phase 5b. (If the send flow turns out to not exist yet, this shrinks to "add a 'Freeze composition' admin action" — decided during implementation based on current state of send code.)

5. **Editing verified invoices requires head_of_procurement approval.** Not a time-window — a permanent approval requirement from day 1. Reuses `services/approval_service.py` and the `kvota.approvals` table. No new approval machinery.

6. **Bypass logic on invoice creation:**
   - New invoice from a supplier who *already* has an invoice on this quote → skip logistics pre-fill AND customs pre-fill (both were presumably filled on the first invoice from that supplier).
   - New invoice from a *new* supplier → skip customs pre-fill only (customs is per-item; logistics differs per-supplier so must be re-entered).

## ONE open decision — confirm before Task 1

**Migration 264 contents.** The brief said "Task 2 → 264" without spelling out what Task 2 does. My proposed interpretation:

```
Migration 264: Column additions to existing tables
  - quote_items.composition_selected_invoice_id UUID NULL
      (FK → kvota.invoices(id) ON DELETE SET NULL;
       the pointer that composition_service reads to pick the active iip row)
  - invoices.verified_at TIMESTAMPTZ NULL
      (when procurement marks an invoice as ready for composition; locks direct edits)
  - invoices.verified_by UUID NULL
      (who verified; references auth.users(id))
```

Migration 263 = the `invoice_item_prices` junction table (with versioning columns from day 1).
Migration 265 = backfill: populate junction from existing `quote_items.invoice_id`, set `composition_selected_invoice_id = invoice_id`, mark `invoices` with status=`completed` as verified.

## Architecture in one diagram

```
                     ┌──────────────────────────────────┐
                     │ Procurement submits supplier KP  │
                     └─────────────┬────────────────────┘
                                   │
                   INSERT invoice_item_prices (one row per item, unfrozen)
                   UPDATE quote_items.invoice_id (legacy pointer — unchanged)
                                   │
                                   ▼
┌──────────────────────────────────────────────────────┐
│  kvota.invoice_item_prices                           │
│  ─────────────────────────                           │
│  (invoice_id, quote_item_id, version) UNIQUE        │
│  purchase_price_original, purchase_currency          │
│  base_price_vat, price_includes_vat                  │
│  frozen_at, frozen_by  ← NULL = editable            │
└──────────────────────────────────────────────────────┘
                                   │
                                   │  Sales opens Calculation step
                                   ▼
              CompositionPicker (frontend/.../calculation-step)
              renders radio per item: Supplier A / Supplier B / ...
                                   │
                    POST /api/quotes/{id}/composition
                   { "item_id": "invoice_id", ... }
                                   │
                                   ▼
           UPDATE quote_items.composition_selected_invoice_id
                                   │
                                   │  Sales clicks Calculate
                                   ▼
 ┌────────────────────────────────────────────────────────────┐
 │ main.py — 3 calc entry points (preview/calculate/??)      │
 │   OLD:  items = supabase.quote_items.select(...)          │
 │   NEW:  items = composition_service.get_composed_items(   │
 │                    quote_id, supabase)                     │
 │                                                            │
 │ composition_service JOINs quote_items with                 │
 │ invoice_item_prices on composition_selected_invoice_id,    │
 │ overlays price/currency/vat fields, returns dicts in       │
 │ EXACT current quote_items shape.                           │
 └────────────────────────────────────────────────────────────┘
                                   │
                                   ▼
                 build_calculation_inputs(items, vars)
                         [UNCHANGED — locked]
                                   │
                                   ▼
                  calculation_engine.calculate(...)
                         [UNCHANGED — locked]
```

**Key architectural insight:** The adapter hook is NOT inside `build_calculation_inputs()` (main.py:12995-13159). It's one layer up — at the three places where `quote_items` is read from the DB and passed into the function (`main.py:13303-13306`, `main.py:14188`, `main.py:14846`). This keeps the surface tiny and the locked files truly untouched.

## Scope of change

### Database (3 migrations)

- `migrations/263_invoice_item_prices.sql` — junction table with versioning, RLS, indexes.
- `migrations/264_composition_pointer_and_verification.sql` — column additions (per the open decision above).
- `migrations/265_backfill_composition.sql` — idempotent backfill.

### Python backend

- **New file:** `services/composition_service.py` — pure service. Key functions:
  - `get_composed_items(quote_id, supabase_client) -> List[Dict]` — the adapter that replaces the 3 quote_items reads.
  - `apply_composition(quote_id, selection: dict, supabase_client) -> None` — validates and persists the user's pick.
  - `get_composition_view(quote_id, supabase_client) -> dict` — returns selections + available alternatives for the picker UI.
  - `freeze_composition(quote_id, user_id, supabase_client) -> None` — stamps `frozen_at` on all current iip rows (called from the send flow, Decision #4).
- **Modifications to `main.py`:**
  - Replace 3 quote_items reads with `composition_service.get_composed_items(...)` calls at lines 13303, 14188, 14846 (verify line 14846 during implementation).
  - Add API routes: `GET /api/quotes/{id}/composition`, `POST /api/quotes/{id}/composition`.
  - Add API routes: `POST /api/invoices/{id}/edit-request`, `POST /api/invoices/{id}/edit-approval/{approval_id}` (reuses approval_service).
  - Add bypass logic to existing invoice-create flow (POST `/api/procurement/{quote_id}/invoices/create` at `main.py:18971-19094`) — detect same-supplier vs new-supplier and skip the appropriate pre-fills.

### Next.js frontend

- **New component:** `frontend/src/features/quotes/ui/calculation-step/composition-picker.tsx` — table UI with per-item radio selection.

## Success criteria

1. **Engine is untouched.** `git diff` on `calculation_engine.py`, `calculation_models.py`, `calculation_mapper.py` shows zero changes at end of Phase 5b.
2. **Backward compatibility.** Every quote that existed before migration 265 still calculates identically after Phase 5b ships (regression test: compute calculation on 5 representative existing quotes before and after backfill, diff results).
3. **Multi-supplier path works end-to-end.** E2E test: create a quote → procurement creates 2 invoices from 2 different suppliers covering overlapping items → sales picks item 1 from supplier A and item 2 from supplier B → Calculate produces a total that matches hand-computation from the selected prices.
4. **Edit-verified approval gates work.** E2E test: mark invoice as verified → try to edit as procurement → see "approval required" modal → submit request → approve as head_of_procurement → edit succeeds.
5. **Bypass logic correct.** Manual test (scripted): create invoice A from supplier X, create invoice B from supplier X (same) → logistics+customs fields NOT pre-filled; create invoice C from supplier Y (new) → logistics IS pre-filled, customs is NOT.
6. **No N+1s.** Composition API returns a single consolidated payload; Python logs show exactly one SELECT for `invoice_item_prices` per composition load.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Backfill race with in-flight procurement work | Migration 265 is idempotent; inserts use `ON CONFLICT DO NOTHING`; runs during a low-traffic window |
| Concurrent composition edits | Optimistic concurrency via `quote_items.updated_at` check on POST `/composition` |
| Orphaned composition pointer if an invoice is deleted | `ON DELETE SET NULL` on `quote_items.composition_selected_invoice_id`; `composition_service` treats NULL pointer as "fall back to `quote_items.invoice_id`" (legacy pointer — Decision #1) |
| Currency mismatch between competing supplier offers | Per-row currency in `invoice_item_prices`; calc pipeline already handles per-item currency at `main.py:13140-13150`. No new logic needed |
| Migration 264 scope ambiguity | The ONE open decision. Blocks Task 1. User confirms before implementation starts |
| Third call site at `main.py:14846` may not exist / may be different | Verified during implementation of Task 5 (hook composition_service into calc flow); grep for `build_calculation_inputs(` to confirm all call sites |
| KP-send flow may not exist yet for Decision #4 freeze hook | If the send flow is a stub, Decision #4 shrinks to "Freeze composition" as an admin action rather than "Freeze on send". Decided at start of the freeze task based on actual current state |
