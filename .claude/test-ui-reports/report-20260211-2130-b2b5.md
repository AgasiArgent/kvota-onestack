**Root Cause (from VPS logs):**
```
Error adding expense to stage: {'message': 'null value in column "planned_amount" of relation "plan_fact_items" violates not-null constraint', 'code': '23502'}
```

The B5 fix correctly changed the code to save expenses with `actual_amount` (факт) instead of `planned_amount` (план). However, the `plan_fact_items` table has a NOT NULL constraint on `planned_amount`. The insert puts the amount in `actual_amount` but leaves `planned_amount` as NULL, violating the constraint.

**Failing row from logs:**
```
(id, deal_id, category_id, description, NULL(planned_amount), RUB, NULL(planned_date), 100.00(actual_amount), RUB(actual_currency), 2026-02-11(actual_date), ...)
```

**Fix needed:** Either:
1. ALTER TABLE to make `planned_amount` nullable: `ALTER TABLE kvota.plan_fact_items ALTER COLUMN planned_amount DROP NOT NULL;`
2. OR set `planned_amount` to 0 (or same as actual_amount) when saving fact expenses

**Additional issue:** The form's date field is named `expense_date` (not `date`), and browser automation tools (Chrome DevTools fill) couldn't properly set the HTML5 date input. The form's required date field validation silently blocked submission when using the UI click method. This is a UX/testing concern but not a code bug.
