## 1. Overview

A new onestack subpage at `/journey` renders every route × role × user-story intersection as an interactive map (React Flow canvas + slide-in drawer), with eight toggleable annotation layers. Its primary users are the development team (P1 — gap analysis, "what exists") and QA (P2 — expected-behaviour pins, verify loop, status). End-user training (P3) and stakeholder demos (P4) are secondary; P4 is explicitly out of scope for v1.0.

The architecture is a **hybrid derived-state + mutable-annotations** split: a CI-generated `journey-manifest.json` captures the immutable truth from the source tree (routes, stories, role gates), and five Supabase tables hold mutable per-node state (impl/qa status, ghost nodes, pins, verifications with screenshot attachments, status history). The two zones join on a stable `node_id` key.

### 1.1 Scope Boundary

**In scope (v1.0):**
- Manifest builder + 3 parsers + CI regeneration check
- 5 new DB tables + migration + RLS policies + `user_has_role` helper
- Python API: 4 endpoints (aggregate read, per-node detail, state PATCH with concurrency, Playwright webhook) + state history endpoint
- Next.js `/journey` page (canvas, drawer, sidebar, 8 layers)
- Pin UI (create, display, verify with screenshot attachments)
- Ghost node CRUD
- Nightly Playwright screenshot pipeline
- Feedback integration via `node_id` column
- Error/loading states for every failure mode
- Testing (unit parsers, integration API, E2E smoke, RLS, concurrency)

**Out of scope (v1.0):**
- Miro/FigJam export (customer demo channel — deferred to v1.1 or later)
- Session recording / replay (potential v2.0)
- Live iframe preview (potential v1.1)
- Mobile layouts
- Customer-facing public journey view
- Analytics on layer usage

## 2. Architecture Pattern & Boundary Map

### 2.1 Pattern: Hybrid Derived-State + Event-Sourced Audit

Two zones with strict immutability boundary:

| Zone | Source | Mutability | Key |
|---|---|---|---|
| **Manifest** (zone A) | `frontend/src/app/**`, `.kiro/specs/**`, `access-control.md` | Immutable; regenerated by CI | Commit hash + `node_id` |
| **Annotations** (zone B) | Human input via UI, Playwright bbox updates | Mutable; per-row audit via history table + append-only verifications | `node_id` |

Joining: every annotation row carries `node_id`. UI loads zone A as static JSON (`frontend/public/journey-manifest.json`) and fetches zone B via API aggregate or Supabase direct.

### 2.3 Import Boundary Enforcement

- Parsers (Node.js TypeScript, `frontend/scripts/journey/**`) SHALL NOT import from `frontend/src/**` — they read files as text, not as modules.
- Next.js page (`frontend/src/app/(app)/journey/**`) imports from: `frontend/src/shared/**`, `frontend/src/entities/journey/**` (new), `@xyflow/react`, public manifest via `fetch`.
- `frontend/src/entities/journey/**` (new FSD slice) owns: types, API client wrappers, React hooks, `canAccessJourney*` guards (per `access-control.md`).
- Python API lives at `api/routes/journey.py` (new file per structure.md parallel-friendly rule; ≤ 800 lines).
- Python business logic for state merges lives at `services/journey_service.py` (new file).

## 3. Technology Stack & Alignment

Aligns with `tech.md`:

| Layer | Choice | Rationale |
|---|---|---|
| Frontend framework | Next.js 15 App Router | Existing stack; page plugs into `(app)` route group |
| Canvas renderer | `@xyflow/react` v12 (formerly `reactflow`) | Industry standard for node-based UIs; native subflows; supports custom nodes, dagre auto-layout via `@dagrejs/dagre` peer |
| Auto-layout | `dagre` | Good-enough for <100 nodes; renders in <50 ms |
| Diff tool (client) | `pixelmatch` + `pngjs` | Standard for client-side image diff; ~30 KB gzipped |
| Data access | Supabase JS (direct CRUD under RLS) + TanStack Query for FastAPI aggregates | Matches existing pattern per `api-first.md` |
| Python API | FastAPI (router in `api/routes/journey.py`) | Matches existing `api/` structure; JWT auth middleware already in place |
| Tests | Vitest (TS), pytest (Python), Playwright (E2E) | Matches existing tooling |
| Storage | Supabase Storage — two buckets: `journey-screenshots` (public read, per-role write), `journey-verification-attachments` (private, signed URLs) | Already used for other uploads; TTL signed URL = 1 hour |

### 3.1 External Dependencies (versions)

- `@xyflow/react` ^12 (package rename from `reactflow`; verified current on npm)
- `@dagrejs/dagre` ^1
- `pixelmatch` ^6, `pngjs` ^7
- `@playwright/test` ^1.50 (already in repo — confirm matching version)

No new runtime dependencies in Python. Existing `supabase-py`, `fastapi`, `httpx` cover the pipeline.

### 3.2 Alignment with Steering

- **`structure.md`**: route under `(app)/journey`, parsers in `frontend/scripts/journey/`, Python in `api/routes/journey.py` + `services/journey_service.py`; 800-line limit respected by splitting frontend into per-layer components under `frontend/src/entities/journey/**` + `frontend/src/features/journey/**`.
- **`tech.md`**: schema `kvota` always; `r.slug` in RLS; pre-commit hook adds manifest to staged files.
- **`access-control.md`**: SELECT for all authenticated users; writes gated per Requirement 12 table; `top_manager` view-only enforced.
- **`api-first.md`**: aggregates and field-aware ACL go through Python; simple single-table CRUD goes through Supabase direct.

History trigger:

```sql
CREATE OR REPLACE FUNCTION kvota.copy_journey_node_state_to_history()
RETURNS trigger AS $$
BEGIN
  INSERT INTO kvota.journey_node_state_history
    (node_id, impl_status, qa_status, notes, version, changed_by, changed_at)
  VALUES
    (OLD.node_id, OLD.impl_status, OLD.qa_status, OLD.notes,
     OLD.version, OLD.updated_by, OLD.updated_at);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER trg_journey_node_state_history
AFTER UPDATE ON kvota.journey_node_state
FOR EACH ROW EXECUTE FUNCTION kvota.copy_journey_node_state_to_history();
```

### 4.4 Python API Contracts (`api/routes/journey.py`)

Endpoint shapes declared via Pydantic v2 models.

```python
# api/models/journey.py (new)
from typing import Literal
from pydantic import BaseModel, Field

NodeId = str  # pattern: r"^(app:|ghost:).+"
ImplStatus = Literal["done", "partial", "missing"]
QaStatus = Literal["verified", "broken", "untested"]

class NodeListItem(BaseModel):
    node_id: NodeId
    route: str
    title: str
    cluster: str
    roles: list[str]
    impl_status: ImplStatus | None
    qa_status: QaStatus | None
    stories_count: int
    feedback_count: int
    pins_count: int

class NodeListResponse(BaseModel):
    success: Literal[True] = True
    data: list[NodeListItem]

class NodeDetailResponse(BaseModel):
    success: Literal[True] = True
    data: "NodeDetail"

class StatePatchRequest(BaseModel):
    impl_status: ImplStatus | None = None
    qa_status: QaStatus | None = None
    notes: str | None = None
    version: int = Field(..., description="Last version client saw; concurrency guard")

class StatePatchResponse(BaseModel):
    success: Literal[True] = True
    data: "JourneyNodeStateDTO"

class StalenessError(BaseModel):
    success: Literal[False] = False
    error: dict  # {"code": "STALE_VERSION", "message": "..."}
    data: "JourneyNodeStateDTO"  # current server state for client refresh

class PlaywrightWebhookPayloadItem(BaseModel):
    pin_id: str
    x: int | None = None
    y: int | None = None
    width: int | None = None
    height: int | None = None
    selector_broken: bool

class PlaywrightWebhookRequest(BaseModel):
    items: list[PlaywrightWebhookPayloadItem]
```

Endpoints (router prefix `/api/journey`):

| Method | Path | Purpose | ACL | Refs |
|---|---|---|---|---|
| GET | `/nodes` | Aggregated canvas data | any authenticated | Req 6.x through 11 |
| GET | `/node/{node_id}` | Full detail (state + pins + verifications latest + training steps + feedback page-1) | any authenticated | Req 5.x |
| PATCH | `/node/{node_id}/state` | Field-aware state update with `If-Match` semantics via `version` | admin / head_of_* for impl; admin / quote_controller / spec_controller for qa | Req 6.1–6.6 |
| GET | `/node/{node_id}/history` | Node state history (≤ 50 rows) | any authenticated | Req 6.9 |
| POST | `/playwright-webhook` | Batched bbox update | `X-Journey-Webhook-Token` header | Req 10.7, 10.8 |

Every handler docstring follows `api-first.md` format (Path, Params, Returns, Side Effects, Roles).

Standard envelope `{ success, data?, error? }` per Req 16.4.

## 5. Key Data Flows

### 5.1 Build-time manifest regeneration

1. Developer edits `frontend/src/app/**`, `.kiro/specs/**`, or `access-control.md`.
2. Husky pre-commit hook detects change via glob; runs `npm run journey:build` in `frontend/`.
3. `build-manifest.ts` invokes the 3 parsers in parallel, merges outputs, writes JSON.
4. Hook `git add`s the regenerated manifest; commit proceeds.
5. CI step on the PR re-runs `npm run journey:build`; `diff` against committed; fails build on mismatch.

### 5.2 Runtime page load

1. User navigates to `/journey`. Middleware verifies auth (redirects if unauth).
2. Server Component fetches manifest from `public/journey-manifest.json` and parses URL state (`node`, `layers`, `viewas`).
3. `GET /api/journey/nodes` fetches aggregate (manifest + state + counts) — TanStack Query cached.
4. Canvas renders via React Flow with dagre auto-layout on first mount; positions cached in state for session.
5. User clicks node → drawer opens → `GET /api/journey/node/{node_id}` populates detail.
6. URL state synced via `router.replace` as user interacts.

### 5.3 Status edit with concurrency guard

1. User edits impl_status in drawer. UI dispatches `PATCH /api/journey/node/{node_id}/state` with current `version`.
2. API loads current row; if `stored.version != request.version` → 409 with current state payload.
3. UI shows toast "Record was updated by another user — refreshing", replaces local state, reverts the user's edit.
4. If versions match, API performs `UPDATE journey_node_state SET impl_status = ..., version = version + 1, updated_at = now(), updated_by = auth.uid()`.
5. Trigger copies old row to `journey_node_state_history`.
6. API responds with new state; TanStack Query cache invalidated.

### 5.4 Nightly Playwright capture

1. GitHub Action fires at 03:00 UTC.
2. Builds frontend, spins up docker-compose (staging DB).
3. For each of 12 roles, logs in as `qa-{slug}@kvotaflow.ru`.
4. For each node visible to that role, navigates, screenshots, uploads to Storage.
5. For each pin on that node, `page.locator(pin.selector).boundingBox()`. Collects batch.
6. After full sweep, POSTs batch to `/api/journey/playwright-webhook` with shared-secret header.
7. API `UPDATE`s pins; marks `selector_broken` where bbox missing.
8. Retention step deletes screenshots older than 2 most recent per `(role, node_id)`.

### 5.6 Journey Flow playback

1. User opens `/journey` and sees the "Пути" panel listing flows grouped by persona role.
2. Click on a flow → navigation to `/journey/flows/:slug`; Server Component fetches the flow from `kvota.journey_flows` by `slug` and the manifest as usual.
3. View renders three panes: step list on the left (current highlighted), focus-node in the centre (same React Flow custom node component used on the main canvas, rendered full-sized), annotated-screen on the right if step's node has pins.
4. User advances via Next button or `→` key. The previous/next steps may focus the same node with different step context (action/note).
5. If a step's `node_id` is missing from the manifest (e.g., ghost node not yet merged), the step still renders with "Узел недоступен" badge — user can skip.
6. Progress counter shows "Step N / M"; estimated remaining time = `flow.est_minutes × (steps_remaining / steps_total)`.
7. Esc returns to canvas with the last-focused node pre-selected.

### 5.5 Verification with screenshot attachment

1. User clicks "broken" on a QA pin, adds note, attaches 1–3 PNGs.
2. Client validates file count, MIME, size.
3. Client uploads to Supabase Storage bucket `journey-verification-attachments` at `{node_id_safe}/{client-generated-uuid}/{index}.png`.
4. On all upload successes, client INSERTs `journey_verifications` row with `attachment_urls` containing the uploaded object keys.
5. RLS allows INSERT for QA roles.
6. On server trigger (optional, or client-side), create linked `kvota.user_feedback` row with the first attachment as the feedback's image.
7. On partial upload failure, all uploaded objects are deleted client-side before aborting the verification.

## 6. Error Handling & Degradation

Per Requirement 14.

| Failure | Detected by | Response |
|---|---|---|
| Manifest fetch fails | `fetch` error | `<ManifestError>` component with retry; canvas blank |
| API 5xx on `/nodes` | TanStack Query retry exhausted | Non-blocking toast + keep stale data visible |
| Empty `journey_node_state` | Aggregate returns null impl/qa | Render grey dots, no error |
| Pin `last_x IS NULL` | Detail response field | Pin in list, no screen overlay |
| Missing screenshot for `(role, node)` | Supabase Storage 404 | Placeholder + "Request capture" (admin only) |
| Expired signed URL | Image load error | Broken-image icon, metadata still renders |
| Optimistic PATCH 409 | Response code | Toast + rollback |
| Optimistic PATCH 403 | Response code | Toast "no permission" + rollback |
| Playwright webhook fails 3 nights | Action logic | Auto-open GitHub Issue `journey-ops` |

## 8. Testing Surface

### 8.1 Unit — parsers (Vitest)

Fixture-based. One fixture per Next.js convention type: plain, `[id]`, `(group)`, `[[catchAll]]`, `@parallel`, interceptor. `build-manifest.test.ts` takes a combined fixture and snapshots the output JSON.

### 8.2 Integration — API (pytest)

Follow existing `tests/test_api_*.py` pattern. Key cases:
- Nodes aggregate joins state + counts correctly (new test_api_journey.py)
- Node detail includes pins and training
- PATCH state returns 409 when `version` mismatches
- PATCH state returns 403 when role lacks field permission
- History endpoint returns rows in reverse chrono
- Webhook rejects missing/invalid token
- Webhook batch updates pins and flags broken selectors

### 8.3 E2E — happy path (Playwright)

Single spec file exercising: open `/journey` as admin → toggle Impl layer → click node → open drawer → mark QA verify with attached screenshot → open another node → create ghost → observe it on canvas.

### 8.4 RLS tests

Matrix test: for each role × each write op, assert allow/deny. `top_manager` gets a dedicated test to confirm read-only.

### 8.5 Concurrency test

Spawn two concurrent PATCH requests with same `version` → assert exactly one 200 and one 409.

## 10. Risks

Extracted to `research.md` §Risks. Summary:

- **Selector rot** — mitigated by `selector_broken` flag + admin-visible "needs attention" list + encouragement of `data-testid`.
- **W5 integration burst** — three new integrations (Playwright auth, Storage, bbox webhook) simultaneously; W6 acts as buffer.
- **32 existing specs without `related_routes:`** — fuzzy-match fallback + one-time backfill script with human review.
- **React Flow perf at 50+ nodes** — viewport culling + memoized custom nodes; loadtest at W2.
- **Storage cost at 12 roles × 34 nodes** — retention of 2 images per pair caps at ~82 MB; verification attachments add ~800 MB saturation.
