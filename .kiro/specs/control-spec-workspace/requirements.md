# Требования — control-spec-workspace

Экран контроля спецификации (двухфазный: «На контроле» → «На подписании») + воркспейс
контроля с двумя канбанами. Цель — дать контролёру спецификаций (`spec_controller`)
нормальный рабочий процесс, **не создавая новой модели данных** (всё уже в БД) и
**не трогая калькулятор**. Контроль расчёта (`quote_controller`) получает вход через
канбан, но его экран не меняется.

> EARS: `Система ДОЛЖНА …` (везде), `КОГДА … Система ДОЛЖНА …` (событие),
> `ЕСЛИ … ТО Система ДОЛЖНА …` (условие), `ПОКА … Система ДОЛЖНА …` (состояние),
> `ГДЕ … Система ДОЛЖНА …` (контекст роли). «Система» = OneStack (Next.js UI + Python API).

## Контекст текущего состояния

- Рельс табов КП (`quote-status-rail.tsx`): Заявка → Закупки → Логистика → Таможня →
  Расчёт → **Контроль** (`control`, статусы `pending_quote_control`/`pending_approval`)
  → Переговоры → **Спецификация** (`specification`, статусы `pending_spec_control`/
  `spec_draft`/`spec_signed`) — финальный таб.
- Таб `control` обслуживается `control-step.tsx` (дашборд 7 проверок). Таб
  `specification` — `specification-step.tsx`.
- `specifications` (m006) уже имеет ~25 полей; UI пишет только `contract_id`,
  `sign_date`, `readiness_period`, `status`.
- В `specification-step.tsx` уже есть: экспорт PDF/DOCX, загрузка скана в storage
  (`kvota-documents`) + запись в `documents`, создание сделки через
  `confirmSignatureAndCreateDeal` → `POST /api/deals` (+7 стадий логистики).
- `customer_contracts` (m022): клиент → много договоров, атомарный
  `get_next_specification_number()`; договоры заводятся на карточке клиента.
- Паттерн воркспейс-канбана: `/workspace/logistics`, `/workspace/customs` (`KanbanPage`
  с props `domain`, dnd-kit).
- Доступ (`.kiro/steering/access-control.md`): `spec_controller`/`quote_controller` =
  тир `FULL_SCOPED_EDIT` (видят всё, редактируют только control/approval-поля своего
  домена); `admin` = full edit; `top_manager` = read-only.

## Требование 1 — Экран контроля спецификации, блок «Из расчёта» (read-only)

**User story:** Как контролёр спецификаций, я хочу видеть ключевые финансовые
результаты расчёта в одном месте, не имея возможности их редактировать, чтобы сверять
спецификацию с утверждённым расчётом без риска создать второй источник правды.

### Acceptance Criteria
1. ГДЕ открыт таб «Спецификация» для КП в статусе `pending_spec_control`, Система ДОЛЖНА
   отображать блок «Из расчёта» со значениями: валюта КП, итого сумма с НДС, профит
   чистый, наценка %, курс на дату КП.
2. Система ДОЛЖНА брать значения блока «Из расчёта» из существующих полей quote/расчёта
   (канонические `*_quote_currency`/`total_with_vat_quote` и т.п.) и отображать их
   строго read-only.
3. Система НЕ ДОЛЖНА предоставлять способ редактирования значений блока «Из расчёта» с
   экрана контроля (никакого дублирующего ввода).
4. ЕСЛИ для КП отсутствует завершённый расчёт, ТО Система ДОЛЖНА показывать в блоке
   понятную пометку «нет данных расчёта» вместо пустых/нулевых значений.

## Требование 2 — Блок «Реквизиты»: выбор из реестров (searchable)

**User story:** Как контролёр, я хочу выбирать наше юрлицо, юрлицо клиента, договор и
страны из существующих реестров через поиск, чтобы реквизиты спецификации были
ссылками на источники истины, а не свободным текстом.

### Acceptance Criteria
1. Система ДОЛЖНА отображать в блоке «Реквизиты» поля: наше юрлицо, юрлицо клиента,
   договор, страна забора груза, страна товара к отгрузке, страна оплаты поставщику.
2. Каждое поле блока «Реквизиты» ДОЛЖНО быть **searchable**-дропдауном (проектный
   стандарт: все селекты с поиском, без исключений).
3. Система ДОЛЖНА наполнять дропдаун «наше юрлицо» из реестра `seller_companies`
   (организации в скоупе `organization_id`).
4. Система ДОЛЖНА наполнять дропдаун «договор» только договорами клиента этой КП
   (`customer_contracts` где `customer_id` = клиент КП и `status = 'active'`).
5. Система ДОЛЖНА наполнять дропдауны стран из справочника стран/локаций.
6. КОГДА у клиента ноль договоров, Система ДОЛЖНА предоставлять инлайн-создание договора
   с экрана контроля (как сейчас в `specification-step.tsx`), создавая запись в
   `customer_contracts`.
7. КОГДА контролёр выбирает значение в любом дропдауне «Реквизитов», Система ДОЛЖНА
   сохранять ссылку (FK), а не отображаемый текст.

## Требование 3 — Блок «Условия спецификации» (ввод)

**User story:** Как контролёр, я хочу заполнять условия спецификации (сроки, тип груза,
город доставки), чтобы спецификация содержала полные договорные условия.

### Acceptance Criteria
1. Система ДОЛЖНА отображать редактируемые поля: дата подписания, срок действия, срок
   готовности, срок на логистику, тип груза, город доставки в РФ — маппинг на
   существующие колонки `specifications`.
2. ГДЕ роль пользователя НЕ имеет права редактировать спецификацию, Система ДОЛЖНА
   показывать эти поля read-only.
3. КОГДА контролёр сохраняет черновик, Система ДОЛЖНА персистить введённые значения в
   `specifications` без смены статуса воркфлоу.
4. Система НЕ ДОЛЖНА требовать поля, дублирующие расчёт (валюта, суммы, наценка) в этом
   блоке — они приходят из блока «Из расчёта».

## Требование 4 — Блок «Контроль»: ответственный, дата контроля, курс на подписание

**User story:** Как контролёр, я хочу зафиксировать себя ответственным за корректность
и проставить дату контроля, а также при необходимости уточнить курс на дату подписания,
чтобы контроль был прослеживаемым, а курсовая разница на подписании — учтена.

### Acceptance Criteria
1. Система ДОЛЖНА записывать текущего пользователя как «ответственного контролёра» при
   действии «Отправить на подписание» (контролёр = ответственное лицо за корректность,
   **не** создатель — спецификацию инициирует МОП в spec-step, как сейчас).
2. КОГДА выполняется «Отправить на подписание», Система ДОЛЖНА проставлять «Дату
   контроля» серверным временем (не хардкод).
3. Система ДОЛЖНА показывать «курс на подписание» как **выбор режима**, по умолчанию
   «по курсу ЦБ на день оплаты» (плавающий, без фиксированного числа), с альтернативой
   «фиксированный курс» (с вводом числового значения). Бизнес-реальность: чаще оплата
   по курсу ЦБ на день оплаты, реже — фиксированный курс.
4. ЕСЛИ выбран режим «фиксированный курс», ТО Система ДОЛЖНА сохранять введённое
   значение в **отдельной** колонке `specifications` (НЕ `exchange_rate_to_ruble`),
   вместе с признаком режима; и НЕ ДОЛЖНА изменять курс в расчёте/калькуляторе.
5. Данные (аддитивно, в рамках Требования 8): новая колонка режима курса
   (`signing_fx_mode`: `cbr_on_payment_day` по умолчанию | `fixed`) и новая колонка
   значения (`signing_fx_rate` NUMERIC, nullable, используется только при `fixed`).

## Требование 5 — Переход «Отправить на подписание» (фаза «На контроле» → «На подписании»)

**User story:** Как контролёр, я хочу одной кнопкой подтвердить корректность данных и
перевести спецификацию на этап подписания.

### Acceptance Criteria
1. ГДЕ спецификация в фазе «На контроле», Система ДОЛЖНА показывать основное действие
   «Отправить на подписание».
2. КОГДА контролёр нажимает «Отправить на подписание», Система ДОЛЖНА перевести
   спецификацию в фазу «На подписании» (статус подписания) и зафиксировать
   ответственного + дату контроля (Требование 4).
3. ЕСЛИ обязательные реквизиты (например, договор и наше юрлицо) не выбраны, ТО Система
   ДОЛЖНА заблокировать переход и явно назвать недостающие поля + визуально их
   подсветить (без «тихих» отказов).
4. Переход воркфлоу ДОЛЖЕН выполняться через Python API (api-first), а не прямым
   мульти-табличным write из Server Action.

## Требование 6 — Фаза «На подписании»: загрузка скана + структурная сверка

**User story:** Как контролёр, я хочу загрузить подписанный скан и пошагово сверить его
с данными в системе, чтобы пометить спецификацию подписанной только при полном
совпадении.

### Acceptance Criteria
1. ГДЕ спецификация в фазе «На подписании», Система ДОЛЖНА предоставлять загрузку
   подписанного скана (переиспользуя существующую загрузку в `kvota-documents` +
   запись в `documents`).
2. Система ДОЛЖНА отображать структурную сверку «скан ↔ данные системы»: пошаговый
   чек-лист / сайд-бай-сайд по ключевым полям (номер спецификации, договор, юрлица,
   суммы, сроки, подписант).
3. ПОКА не подтверждены все пункты сверки, Система ДОЛЖНА держать кнопку «Пометить
   подписанной» неактивной.
4. ЕСЛИ подписанный скан не загружен, ТО Система ДОЛЖНА держать сверку и «Пометить
   подписанной» недоступными.
5. Система ДОЛЖНА фиксировать факт прохождения сверки (кто и когда подтвердил) для
   прослеживаемости.

## Требование 7 — Завершение цепочки: «Пометить подписанной» → воркфлоу сделки

**User story:** Как контролёр, я хочу пометить спецификацию подписанной, чтобы цепочка
КП завершилась и началась сделка.

### Acceptance Criteria
1. КОГДА все пункты сверки подтверждены и нажата «Пометить подписанной», Система ДОЛЖНА
   вызвать существующий `confirmSignatureAndCreateDeal` → `POST /api/deals` (создание
   сделки + 7 стадий логистики); эта логика НЕ переписывается.
2. КОГДА сделка создана, Система ДОЛЖНА перевести КП в терминальный статус `deal`
   (цепочка табов завершена).
3. Система ДОЛЖНА считать воркфлоу сделки (после создания) вне скоупа — обеспечивается
   только корректная передача.
4. ЕСЛИ создание сделки завершилось ошибкой, ТО Система ДОЛЖНА показать ошибку и
   сохранить спецификацию в фазе «На подписании» (без частичного/«тихого» перехода).

## Требование 8 — FK-миграция реквизитов (expand-contract)

**User story:** Как разработчик/владелец данных, я хочу, чтобы ссылочные поля
спецификации стали FK на реестры, чтобы убрать второй источник правды и опечатки
свободного текста.

### Acceptance Criteria
1. Система ДОЛЖНА хранить наше юрлицо / юрлицо клиента / страны на `specifications`
   как FK на соответствующие реестры: наше юрлицо → `seller_companies`, страны →
   `locations` (решение владельца на текущем этапе; пересмотрим, если появится
   отдельный справочник стран). Дополнительно добавляются колонки курса на подписание
   из Требования 4.5.
2. Миграция ДОЛЖНА быть аддитивной (expand-contract): новые FK-колонки добавляются, не
   ломая существующие данные; старый свободный текст бэкфиллится сопоставлением.
3. Миграция ДОЛЖНА следовать проектным правилам: схема `kvota.*`, последовательная
   нумерация (после 283), `BEGIN/COMMIT` для мульти-statement, применение на проде через
   `git pull` + `scripts/apply-migrations.sh` (НЕ scp) **до** мержа PR, читающего новые FK.
4. КОГДА миграция применена, Система (процесс разработки) ДОЛЖНА перегенерировать типы
   фронтенда (`npm run db:types`) и пройти `tsc` зелёным до любого drop-а старых колонок.
5. Система НЕ ДОЛЖНА изменять `calculation_engine.py` / `calculation_models.py` /
   `calculation_mapper.py`.

## Требование 9 — Воркспейс `/workspace/control` с двумя канбанами

**User story:** Как контролёр, я хочу единое рабочее место со своей очередью, чтобы не
заходить в каждую КП по отдельности.

### Acceptance Criteria
1. Система ДОЛЖНА предоставлять страницу `/workspace/control` по паттерну логистики
   (`KanbanPage`/dnd-kit) с двумя канбанами и переключателем между ними: «Контроль
   расчёта» и «Контроль спецификации».
2. Канбан «Контроль расчёта» ДОЛЖЕН показывать КП в статусах `pending_quote_control` и
   `pending_approval`.
3. Канбан «Контроль спецификации» ДОЛЖЕН показывать спецификации в фазе контроля, с
   колонками «На контроле» (`pending_spec_control`) и «На подписании» (фаза подписания).
4. Система ДОЛЖНА добавить пункт «Контроль» в сайдбар, ведущий на `/workspace/control`.
5. Система ДОЛЖНА фильтровать содержимое канбанов по `organization_id` до любых других
   фильтров.
6. Возвраты на доработку (уходящие в `client_negotiation`) Система НЕ ДОЛЖНА показывать
   как отдельную колонку канбана контроля (они покидают этап контроля).

## Требование 10 — Навигация: карточка канбана → нужный таб КП

**User story:** Как контролёр, я хочу из карточки канбана попадать сразу на нужный шаг
карточки КП.

### Acceptance Criteria
1. КОГДА контролёр кликает карточку в канбане «Контроль расчёта», Система ДОЛЖНА открыть
   `/quotes/{id}?step=control`.
2. КОГДА контролёр кликает карточку в канбане «Контроль спецификации», Система ДОЛЖНА
   открыть `/quotes/{id}?step=specification`.
3. Существующий таб «Контроль» (расчёта, `control-step.tsx`) Система НЕ ДОЛЖНА изменять
   по содержанию — добавляется только вход через канбан.

## Требование 11 — Доступ и видимость (RLS / тиры ролей)

**User story:** Как владелец продукта, я хочу, чтобы каждый видел и редактировал только
своё, чтобы соблюсти модель доступа.

### Acceptance Criteria
1. ГДЕ роль = `quote_controller`, Система ДОЛЖНА показывать канбан «Контроль расчёта»;
   ГДЕ роль = `spec_controller`, Система ДОЛЖНА показывать канбан «Контроль
   спецификации»; ГДЕ роль = `admin`/`top_manager`, Система ДОЛЖНА показывать оба.
2. ГДЕ роль = `top_manager`, Система ДОЛЖНА делать экран контроля read-only.
3. Система ДОЛЖНА разрешать редактирование полей контроля спецификации только ролям
   `spec_controller`/`admin` (control/approval-поля своего домена, тир
   `FULL_SCOPED_EDIT`).
4. Детальные экраны ДОЛЖНЫ защищаться `canAccess*`-гардом и возвращать `notFound()` при
   отказе (никогда `403`, не утекать существование).
5. ЕСЛИ роль неизвестна/неоднозначна, ТО Система ДОЛЖНА отказывать в доступе (fail
   closed). Проверки ролей — только через `shared/lib/roles.ts`, без инлайна.

## Требование 12 — Нефункциональные ограничения

### Acceptance Criteria
1. Система НЕ ДОЛЖНА создавать новую модель данных для спецификаций/договоров — все поля
   уже существуют (`specifications` m006, `customer_contracts` m022).
2. Система НЕ ДОЛЖНА перестраивать рельс табов и НЕ ДОЛЖНА вводить новые статусы
   воркфлоу сверх уже существующих (`pending_spec_control`, фаза подписания, `deal`).
3. Все переходы воркфлоу и мульти-табличные writes ДОЛЖНЫ идти через Python API
   (api-first); Server Actions — тонкие обёртки (auth → API → revalidate).
4. Система ДОЛЖНА переиспользовать существующие экспорт PDF/DOCX, загрузку скана и
   создание сделки — без переписывания.
5. Все хардкоды запрещены: даты — серверным временем, URL/ID — из конфигурации/данных.

## Зафиксированные архитектурные решения (после gap-анализа, 2026-05-29)

Эти решения разрешают ключевые флаги `gap-analysis.md` и обязательны для фазы дизайна:

1. **Реактивировать `pending_signature`.** Сейчас `create_deal` ставит
   `workflow_status='deal'` напрямую, минуя `pending_signature` (статус мёртв в проде).
   Надо: «Отправить на подписание» → `callWorkflowTransition(to_status='pending_signature')`;
   deal-handoff («Пометить подписанной») гнать через `workflow_service.transition_quote_status`
   (`pending_signature → deal`, с аудитом и проверкой ролей), а НЕ прямой записью в
   `api/deals.py`. Колонка канбана «На подписании» = `pending_signature`. Spec-экран
   гейтит на `quotes.workflow_status` (единый источник истины воркфлоу), не плодим
   третий writer поверх `specifications.status`.
2. **Юрлицо клиента — один набор реквизитов, read-only** (из `customers`/snapshot).
   Полноценный мульти-юрлицо дропдаун (новая таблица `customer_legal_entities`)
   **отложен** — вне скоупа.
3. **Структурная сверка (Req 6) — ручной чек-лист**: контролёр глазами сверяет скан с
   данными системы и подтверждает пункты. **Без OCR / без content-equality** —
   подтверждается существование скана + совпадение метаданных (номер спеца, договор,
   юрлица, суммы, сроки, подписант).
4. **Канбан — кликабельные карточки** (клик → нужный таб карточки КП). **Без
   drag-to-action**; действие контролёра — на экране шага.
5. **Воркспейс — один route `/workspace/control`** с переключателем между двумя
   канбанами (не два route). Fail-closed гард на странице (redirect/notFound для
   неавторизованных — сознательная дивергенция от fail-open прецедента logistics/customs).
6. **Реквизиты persist как FK** (`our_legal_entity` → `seller_companies`, страны →
   `locations`) — но **сначала проверить в дизайне**, что эти колонки не читаются
   калькулятором (`build_calculation_inputs()`); если читаются — оставить free-text
   snapshot, чтобы не нарушить «calc-engine LOCKED».
7. **Блок «Из расчёта»** читать `total_quote_currency` (НЕ удалённый `total_amount_quote`),
   `total_with_vat_quote`, `total_profit_usd`, `currency`, `exchange_rate_to_usd`.
8. **Номер миграции** подтвердить на VPS (`kvota.schema_migrations`) до написания ALTER
   (CLAUDE.md=283 расходится с `migrations/`=318/319).

См. полный список флагов и точек интеграции в `gap-analysis.md`.
